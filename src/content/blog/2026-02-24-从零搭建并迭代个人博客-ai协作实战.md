## 0. 项目背景：不是“做个站”，而是“建立长期写作系统”

这次的真实目标并不是做一个好看的 demo，而是搭建一个能长期写作、可维护、低运维成本的博客系统。

约束条件很明确：

- 不想上复杂云服务（避免后续维护负担）
- 希望视觉现代、但不牺牲性能
- 旧博客有内容资产，需要尽可能迁移
- 希望借助 AI 提升交付速度

所以我们最终采用：

**Astro + Markdown + GitHub Pages + GitHub Actions**

这是一条非常“个人项目友好”的路线：

- 成本低
- 攻击面小（纯静态）
- 迭代快
- 可持续（内容就是仓库里的 Markdown）

---

## 1. 技术决策：为什么是 Astro

### 对比思路

我最开始考虑过三种方向：

1. 传统动态博客（数据库 + 后台）
2. 纯静态站点生成器（SSG）
3. 本地写作 + 云端托管静态产物

结合“低维护 + 安全 + 可迭代”，最终选了第 3 种。

### Astro 的关键优势

- 默认输出静态页面，性能和部署都简单
- 内容结构清晰，Markdown 组织成本低
- 主题和样式改造空间大，适合后期做品牌化

结论：**Astro 很适合“长期个人内容系统”而不只是“新手练手项目”。**

---

## 2. 首次上线：从可用到可访问

### 初始化与部署

第一步很顺利：

- 初始化 Astro blog 模板
- 配置 GitHub Actions 自动构建
- 推送到 GitHub Pages

但上线后立刻遇到经典问题：

- 首页可访问，`/blog` 404
- 文章详情页 404

### 根因定位

不是构建失败，而是 **GitHub Pages 子路径部署** 的路径策略问题。

仓库站点实际部署在：

`https://aomeilsh-maker.github.io/personal-blog`

这意味着所有内部链接都必须考虑 `base=/personal-blog`。

### 修复要点

1. `astro.config.mjs` 配置 `base`
2. 导航链接统一走 `BASE_URL`
3. 列表页到详情页的 href 统一加子路径前缀

这是本次项目第一个关键经验：

> 静态站上线后“看起来像路由问题”的 404，80% 是路径策略没统一。

---

## 3. 内容迁移：从“抓取失败”到“导出导入”

### 原计划：直接抓旧站正文

我们先尝试从旧站 archive 进入文章链接抓正文。

结果：

- archive 可读，但正文接口不可用
- 旧站背后的 Notion 数据访问异常
- 页面可打开但内容源为空壳

### 改用稳定路径：Notion Markdown 导出

后续改成：

1. Notion 导出 Markdown 包
2. 批量映射文章 ID 与导出文件
3. 生成/覆盖新站文章 Markdown
4. 保留标题、日期、分类等元信息

这条链路稳定完成了存量迁移。

第二个关键经验：

> 迁移时优先“官方导出格式”，不要过度依赖线上页面抓取。

---

## 4. 视觉迭代：从模板感到内容品牌感

### 第一轮（基础可读）

- 首页重写（去掉模板默认文案）
- Blog 列表改为卡片
- 详情页排版优化（标题区/段落/表格/引用）

### 第二轮（结构升级）

- 首页加入“精选文章”封面流
- 分类页从“纯链接”升级为“可视化分类卡片”
- 列表卡片加入封面图、分类徽章、信息层级

### 第三轮（统一风格）

- 三类内容配置分类主题图
- 统一卡片视觉语言（阴影、悬浮、边框、留白）
- 完善深浅色调兼容

第三个关键经验：

> 博客风格升级不是“换一套 CSS”，而是“信息结构 + 视觉层级 + 内容语义”一起改。

---

## 5. 分类系统：让博客从“时间线”变成“知识空间”

后期我们固定了三大分类：

- 日语学习
- AI学习
- 杂文

并批量把历史文章归类，配套完成：

- 分类入口
- 分类聚合页
- 列表与详情的分类可视化

意义在于：

- 读者能按主题浏览
- 后续可扩展标签筛选、专题页、推荐流

这一步是“个人博客工程化”的分水岭。

---

## 6. AI 在这次项目中的角色：高效执行，不替代判断

本次协作里，AI 的价值主要在：

- 快速生成改造代码
- 快速试错与修复
- 大量重复性修改（路径、样式、迁移脚本）

但仍需要人做三件事：

1. 给出清晰目标（审美/策略/边界）
2. 验收关键结果（功能是否真的可用）
3. 决定取舍（速度 vs 质量，简洁 vs 表达）

我更认可的协作模式是：

> 人负责方向与判断，AI负责实现与加速。

---

## 7. 复盘：这次做对了什么

### 做对的点

- 一开始就选了低维护架构
- 每次改动都走了“构建验证 -> 部署验证”闭环
- 迁移失败后快速切换到更稳定方案
- 没停在“能用”，而是持续迭代到“好用 + 好看”

### 可以更好的点

- 早期可以更早统一 URL 策略，减少 404 返工
- 文章 frontmatter 规范可在迁移前先定义完整
- 图片素材可以提前做版权可追踪的素材池

---

## 8. 给想做个人博客的你

如果你也想快速起站，我建议按这个顺序：

1. 先打通发布闭环（写 -> 构建 -> 部署）
2. 再做迁移与分类
3. 最后做视觉品牌化

不要反过来。

因为博客真正的壁垒，不是首屏多漂亮，而是能不能持续输出。

---

## 结语

这次项目的价值，不只是一套页面，而是一条可以反复使用的协作工作流：

- 明确目标
- 小步迭代
- 快速验证
- 持续重构

当 AI 成为协作伙伴后，个人创作者能把更多精力从“搭环境”转移到“做内容与判断”。

而这，才是效率提升真正发生的地方。
